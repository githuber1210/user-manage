一、什么是 Agent

Agent 不是聊天机器人，而是一个可以自主完成任务的软件系统。

与普通对话不同，Agent 能够：

主动分解复杂任务并逐步执行
调用外部工具（文件系统、数据库、API 等）
在多轮步骤中保持上下文和目标
根据执行结果自主调整后续行动




二、大模型直接使用的痛点

直接将大模型用于生产任务时，通常面临三类核心问题：

问题	表现	影响
概率性输出	相同输入产生不同结果	行为不可预测，难以标准化
幻觉（Hallucination）	自信地给出错误答案	数据错误、逻辑失效
上下文爆炸	加载过多工具/文档导致遵循能力下降	指令失效、关键步骤被忽略

解决方案：通过 AGENTS.md 设置"物理边界"和"全局指令"，通过 SKILL 实现上下文的渐进式披露，从根本上约束 Agent 的行为空间。

三、AGENTS.md 开发规范
什么是 AGENTS.md？

AGENTS.md 是 Agent（如 Trae/ Claude Code / Codex）在执行任何工作之前读取的全局指令文件。它允许你：

设置全局编码规范与技术约束
定义项目特定的工作流程
为不同目录提供不同的指导
标准化团队的开发实践
编写核心原则
原则	说明
Less is More	只写 AI 无法自学的内容，不写常识
侧重禁止操作	用 Negative Constraints（禁止什么） 明确边界，比描述"应该做什么"更有效
信任推理能力	不要锁死操作步骤，给 AI 留有推理空间
文件极小化	正文内容不超过 200 行

# Agent 定义：资深后端开发专家

## 技能限定 (Skills Scoping)
- **业务技能调用限制**: 
  - 严禁调用任何以 jmbos- 开头但不在 skills/jmbos-backend* 路径下的技能。
- **通用技能开放原则**: 
  - 除 `jmbos-` 前缀外的所有授权技能（如 git, file_operator, db_audit 等）均可按需调用。
  - 仅限执行已获得显式安全授权的自动化脚本。

## 核心准则 (Operation Rules)
1. **环境兼容性**: 严格遵循 JDK 8 规范，禁止引入 JDK 9+ 语法（var、List.of、接口私有方法等）。
2. **依赖变更**: 未获得显式授权时，严禁修改 pom.xml 中的依赖版本号。
3. **数据库安全**: 生成 SQL 前必须执行 `skills/db-secure-audit` 自检规则；
   严禁生成不带 WHERE 条件的 UPDATE 或 DELETE 语句。
4. **脚本审计**: 生成任何 Shell 命令前，自检是否含递归删除（rm -rf）或敏感目录修改。

## 项目上下文 (Project Context)
- **全局概览**: 启动任务前，必须首先读取工作区目录下的索引，获取知识
  获取最新的项目背景、架构说明及目录规范。

## 工作流 (Workflow)
1. **自主索引**: 通过 `jmbos-*` 匹配模式定位 Skill 文件，禁止在未检索前询问用户项目结构。
2. **代码分析**: 扫描目标类及依赖工具包（util），优先复用现有公共方法，严禁重复开发已有功能。
3. **最小化修改**: 仅修改必要代码，禁止重写无关样板代码或格式化非相关区域。
4. **验证交付**: 提供自测命令建议，并标注改动对现有接口兼容性的潜在影响。

四、SKILL 规范
为什么需要 SKILL？

当前大模型的主要瓶颈在于上下文长度限制。MCP 工具数量过多时，冗长的系统提示词和工具描述会显著降低模型的指令遵循能力，具体表现为：

模型需要更长时间分析工具调用路径
关键指令被稀释，遵循能力下降
无关 Token 带来延迟和成本浪费

SKILL 的解决思路：采用渐进式披露（Progressive Disclosure），用尽可能少的上下文实现功能感知，真正做到"有效加载上下文"。




SKILL 的四类组件
组件	作用	加载时机
元信息（YAML 元数据）	name + description，唯一标识符	启动时默认加载
主体内容（Body）	概述、入门指导、资源引用目录	Agent 选中该 Skill 后加载
参考文件（Reference File）	详细指南、API 参考、使用示例	Body 明确链接时动态加载
脚本文件（Script）	可执行脚本，保证一致性执行	按需调用




命名规范

启动 SKILL 时只预加载所有 SKILL 的 name 和 description，命名是成功调用的第一步。

推荐格式：动词-ing + 对象

✅ 正例：processing-pdfs
✅ 正例：analyzing-spreadsheets
✅ 正例：managing-databases

❌ 反例：helper        （过于模糊）
❌ 反例：documents     （无动作，不知能做什么）
❌ 反例：utils         （完全无语义）
描述规范

description 是模型选择 Skill 的核心依据，必须做到有区分度，包含：

功能说明：这个 Skill 能做什么

触发条件：何时应该使用它（具体上下文/关键词）

✅ 正例：
description: >
  从 PDF 文件中提取文本和表格，填写表单，合并文档。
  适用于处理 PDF 文件，或用户提及 PDF、表单、文档提取的情况。

❌ 反例（过于模糊）：
description: 处理 pdf 文件

❌ 反例（冗余堆砌）：
description: >
  此技能旨在帮助用户处理 PDF 文档，提供全面的功能，
  包括从 PDF 文件中提取文本内容和表格数据、填写 PDF
  文档中的表单字段以及将多个 PDF 文档合并为一个文件。
  无论您是需要处理 PDF 文件、用户明确表示想要处理
  PDF 文件……（过长，模型注意力分散）

大模型已足够聪明，只需补充它不了解的背景信息，不需要补充常识。

✅ 正例（直接给关键信息）：

## 提取 PDF 文本

使用 pdfplumber 提取文本：

自由度匹配原则

同任务需要赋予不同的执行自由度：

自由度	任务特征	指令设计建议	典型场景
高	多种方法都可成功；决策依赖上下文	给出目标和方向性原则，信任模型选择最优路径	代码审查、方案设计
中	存在推荐执行模式；允许一定变体	固定整体结构，通过参数开放可变部分	带参数的自动化脚本
低	只有一种正确路径；错误代价极高	提供明确护栏和固定执行顺序，禁止模型自行推断	数据库迁移、生产部署

高自由度示例（文本描述即可）：

1. 分析代码结构和组织方式
2. 检查潜在的错误或边界情况
3. 提出提高可读性和可维护性的改进建议
4. 验证是否符合项目规范

中自由度示例（伪代码 + 参数）：

def generate_report(data, format="markdown", include_charts=True):
    # 处理数据
    # 以指定格式生成输出
    # 可选：包含可视化图表

低自由度示例（精确命令，禁止变体）：

## 数据库迁移
请严格执行以下脚本，不得修改命令或添加任何参数：

    scripts/migrate.py --verify --backup




五、技能过多时的优化方案

当 Skill 数量超过 10 个时，推荐放弃全量加载，转为基于命名空间的精确限定策略。

建立路由级名称限定

在 AGENTS.md 中通过通配符显式声明 Agent 的"合法技能"：

## 技能限定
- 业务技能：`skills/jmbos-*`     # 仅加载业务逻辑相关组件
- 通用工具：`skills/common-*.sh` # 仅允许已授权的自动化脚本
动态加载与注意力收敛

黄金区间：通过名称过滤，确保单次对话激活的 Skill 保持在 3–5 个。

这样做的收益：

最大化 Claude 对复杂逻辑的遵循能力
显著降低无效 Token 带来的延迟
防止模型调用无关逻辑导致误操作

skill 总量 > 10  →  按业务线划分命名空间  →  AGENTS.md 中硬性指定加载范围
                                                        ↓
                                          单次对话激活 Skill ≤ 5 个

7.2 构建第一个 SKILL：以"生成工作日报表格"为例




1.详细表达需求进行提问

使用 skill-creator 创建一个名为 daily-report-generator 的 Skill。

1. 目标：

自动化生成标准格式的工作日报 Excel。该 Skill 不读取本地配置文件，完全通过对话获取必要信息。

2. 交互流程（Body 逻辑）：

主动询问：当用户触发该 Skill 时，Agent 必须依次询问以下 6 个关键参数：

年份与月份（如：2026年3月）

岗位（建议默认值：XXXXX）

人员名称（建议默认值：XXX）

系统（建议默认值：XXXXX）

上班时间（建议默认值：9:00:00）

下班时间（建议默认值：18:30:00）

确认执行：在收集完所有信息后，Agent 需列出清单供用户确认，确认无误后调用脚本。

3. 脚本组件 (Script)：

编写 report_engine.py。

核心逻辑：使用 openpyxl 加载 workReport/template.xlsx。

动态计算：集成节假日 API（api.jiejiariapi.com）逻辑，自动计算指定月份的工作日。

数据填充：按照模板样式动态插入行。

表头填充岗位和日期区间。

明细行填充序号、姓名、日期、上下班时间和系统名称。

自动保留模板的边框、字体、对齐方式。

自动生成“合计”行和底部的“签字”区域。

4. 资源要求：

脚本需指向相对路径 workReport/template.xlsx 寻找模板。

生成的文件命名规则：姓名-YYYY年MM月份日报统计.xlsx。

5. 元信息 (YAML)：

name: daily-report-generator

description: 通过对话收集参数，结合本地模板和节假日 API 自动生成定制化工作日报 Excel。

收集：Agent 会根据你给的逻辑，像填表一样询问你：“好的，岗位还是‘JAVA开发工程师’吗？姓名是XXX吗？...”

处理：在你确认后，Agent 会在后台运行我们整合进去的 report_engine.py。
